## Plan: Unified Simulation Task Pipeline

Standardize simulation-task mutations on an MCP-first lifecycle, bias task creation toward executable research and analytics deliverables instead of meeting coordination, treat `@agent` delegation as a pending task-offer flow, require lifecycle changes to be reflected back into the simulation chat, enforce task work against the simulation clock, and package completed deliverables into the final report folder. The product UI should be read-only for task state: simulation agents own task offers, acceptance, progress, blocking, artifact creation, and completion. XML remains a compatibility shim only, and browser-driven task mutations should be removed or hard-gated out of normal runtime usage.

**Task-shape policy**
1. Simulation-created tasks should default to concrete deliverables an assignee can produce with the available MCP tools and current simulation context: research briefs, market scans, KPI summaries, competitor comparisons, risk memos, recommendation notes, evidence packs, or draft report sections.
2. Coordination-only requests such as "set up a meeting", "schedule a sync", or "talk to Marketing" are not first-class executable tasks. The sender or receiver should rewrite them into an actionable deliverable request or decline them with a reason that meetings are out of scope for the simulation runtime.
3. Every accepted task should leave a visible public trail in the simulation chat: offer and acceptance or decline, meaningful progress or blocker updates, and a completion summary that points to the deliverable or inline output.

**Decisions**
- Canonical runtime path: MCP-first for in-simulation task mutation; XML remains a compatibility shim only.
- UI policy: the product UI is read-only for simulation tasks; simulation agents own all lifecycle mutations and task artifacts during a run.
- REST mutation policy: browser-driven task mutation routes are not part of the normal product contract and should be removed or hard-gated.
- Task-shape policy: simulation-task offers should default to tool-executable research and analytics deliverables; coordination-only meeting asks should be rewritten or declined.
- Mention behavior: a public `@agent` delegation should create a pending task offer by default, with the receiver explicitly accepting or declining it.
- Chat follow-up policy: accept/decline, major status updates, and completion should be reflected back into the public simulation chat with deliverable references when possible.
- Report packaging: completed `.md`, `.txt`, and `.pdf` task outputs should be copied into the final report folder and surfaced in Step 4.
- Included scope: task lifecycle, mention-to-offer flow, time-bounded execution, report packaging, and Step 3 / Step 4 / Step 5 UX integration.
- Excluded scope: private/direct messaging between agents, simulating off-screen meetings, browser-side/manual task management, replacing the OASIS runtime model, or building a separate workflow engine outside the existing simulation/report architecture.

**Further Considerations**
1. Default deadline policy: use `due_round = total_rounds` in the first pass unless a sender explicitly sets an earlier target; this keeps the rule easy for agents and the UI to explain.
2. Deliverable policy: treat inline completion summaries as valid lightweight outputs, but only stage/copy files for `.md`, `.txt`, and `.pdf` in the first pass to keep artifact handling bounded.
3. Task-template library: seed prompt examples for research brief, KPI snapshot, competitor analysis, risk memo, evidence pack, and recommendation summary so agents have executable alternatives to meeting language.
4. Rollout safety: keep compatibility-path usage and forbidden manual-path attempts visible in logs and metrics so you can remove legacy XML behavior once agent prompts reliably use the MCP-first tools and confirm the browser can no longer mutate task state.
